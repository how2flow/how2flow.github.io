I realized I haven't done a driver code review yet, so I'm starting one now.<br>
The target is the M1S kernel released by Hardkernel late last year.<br>
I'll be looking at the simple **gpiomem** driver.<br>

## Overview

The **gpiomem driver** is a device driver that isolates only the GPIO region within memory.<br>
When hardware is controlled from userspace, it is typically done with root privileges.<br>

But can we control GPIO through the <span style="{{ site.code }}">/dev/mem</span> node?<br>
It's possible, but if you accidentally access a non-GPIO address and change register values...<br>
Writes performed with root privileges are applied immediately.<br>
While it might not always cause a major issue, it could significantly impact the system.<br>

To prevent such situations, the node is restricted to open only GPIO addresses.<br>
For instance, <span style="{{ site.code }}">wiringPi</span> uses <span style="{{ site.code }}">/dev/gpiomem</span>,<br>
and <span style="{{ site.code }}">gpiod</span> uses <span style="{{ site.code }}">/dev/gpiochip[n]</span> nodes.<br>

There may be other cases, but let's start with the gpiomem driver.<br>

```
static const struct vm_operations_struct rk3568_gpiomem_vm_ops = {
#ifdef CONFIG_HAVE_IOREMAP_PROT
	.access = generic_access_phys
#endif
};

static int address_is_allowed(unsigned long pfn, unsigned long size)
{
	unsigned long address = pfn << PAGE_SHIFT;
	
	dev_info(inst->dev, "address_is_allowed.pfn: 0x%08lx", address);
	
	switch(address) {
		case 0xfdd00000:
		case 0xfdd20000:
		case 0xfdc20000:
		case 0xfdd60000:
		case 0xfdd70000:
		case 0xfe6f0000:
		case 0xfe740000:
		case 0xfe750000:
		case 0xfe760000:
		case 0xfe770000:
		case 0xfdc60000:
			dev_info(inst->dev, "address_is_allowed.return 1");
			return 1;
			break;
		default :
			dev_info(inst->dev, "address_is_allowed.return 0");
				return 0;
    }
}

static int rk3568_gpiomem_mmap(struct file *file, struct vm_area_struct *vma)
{

	size_t size;

	size = vma->vm_end - vma->vm_start;


	if (!address_is_allowed(vma->vm_pgoff, size))
		return -EPERM;

	vma->vm_page_prot = phys_mem_access_prot(file, vma->vm_pgoff,
						 size,
						 vma->vm_page_prot);

	vma->vm_ops =  &rk3568_gpiomem_vm_ops;

	/* Remap-pfn-range will mark the range VM_IO */
	if (remap_pfn_range(vma,
			    vma->vm_start,
			    vma->vm_pgoff,
			    size,
			    vma->vm_page_prot)) {
		return -EAGAIN;
	}

	return 0;
}
```

Among the function parameters is <span style="{{ site.code }}">vm_area_struct</span>.<br>
This is a virtual memory structure, but first, we need to understand what mmap is.<br>

mmap stands for memory mapping.<br>
The kernel performs memory mapping by creating separate virtual addresses,<br>
and these virtual address regions are defined using <span style="{{ site.code }}">vm_area_struct</span>.<br>
<U>Understanding this requires knowledge of virtual memory and paging techniques</U>.<br>
Since memory is not our primary topic, let's move on with the code review.<br>

The size of the VMA is <span style="{{ site.code }}">vma->vm_end - vma->vm_start</span>.<br>
These represent the end and start addresses of the VMA, respectively.<br>
Naturally, subtracting the start address from the end address gives the size of the region.<br>

And there is a custom function in the middle:
```
	if (!address_is_allowed(vma->vm_pgoff, size))
		return -EPERM;
```

As mentioned, **gpiomem** isolates only the GPIO region.<br>
This step verifies the region.<br> It checks if the <span style="{{ site.code }}">vma->vm_pgoff</span> value is within the GPIO region.<br>
<span style="{{ site.code }}">vma->vm_pgoff</span> represents the page offset of the VMA.<br>
This is the starting position of the actual file.<br>
A virtual memory region contains multiple pages, typically 4K to 8K in size.<br>
If you're familiar with paging, you'll know that the actual file start is not the same as the virtual memory start address.<br>
This is called the <span style="{{ site.code }}">Page Frame Number</span>, or <span style="{{ site.code }}">PFN</span> for short.<br>

<br>
This section checks whether the address pointed to by the <span style="{{ site.code }}">file descriptor</span> is indeed within the GPIO region when using **gpiomem**.
```

{

```
It actually receives the pfn as a parameter.<br>
<span style="{{ site.code }}">address</span> is the physical address.<br>
Generally, the operation <span style="{{ site.code }}">pfn << PAGE_SHIFT</span> converts it to a physical address,<br>
conversely, <span style="{{ site.code }}">phy >> PAGE_SHIFT</span> yields the pfn value.<br>
<span style="{{ site.code }}">PAGE_SHIFT</span> typically has a value between 12 and 16.<br>
This value varies depending on the architecture.<br>

The converted address is then compared with the GPIO region addresses.<br>
A <span style="{{ site.code }}">switch-case</span> statement determines whether to proceed with mmap or terminate based on the return value.<br>
The GPIO region can be identified by checking the chip's datasheet.<br>

As mentioned previously, I will not be explaining the virtual memory structure in detail.
```
	if (remap_pfn_range(vma,
```

This is the most critical part of the mmap function.<br>
The <span style="{{ site.code }}">remap_pfn_range</span> function is used for memory mapping.<br>
It creates the page table for the virtual address.<br>
Ultimately, it establishes a mapping between vma->pgoff (the virtual address) and the actual hardware physical address.<br>
The mapping occurs within the VMA region (vm_start to size).<br>
That's the basic idea.<br>
The error log ( <span style="{{ site.code }}">EAGAIN</span> ) corresponds to "No more processes".<br>

Driver definition section:
```
static int rk3568_gpiomem_probe(struct platform_device *pdev)
{
	int err;
	void *ptr_err;
	struct device *dev = &pdev->dev;
	struct resource *ioresource;

	/* Allocate buffers and instance data */

	inst = kzalloc(sizeof(struct rk3568_gpiomem_instance), GFP_KERNEL);

	if (!inst) {
		err = -ENOMEM;
		goto failed_inst_alloc;
	}

	inst->dev = dev;

	ioresource = platform_get_resource(pdev, IORESOURCE_MEM, 0);
	if (ioresource) {
		inst->gpio_regs_phys = ioresource->start;
	} else {
		dev_err(inst->dev, "failed to get IO resource");
		err = -ENOENT;
		goto failed_get_resource;
	}

	/* Create character device entries */

	err = alloc_chrdev_region(&rk3568_gpiomem_devid,
				  DEVICE_MINOR, 1, DEVICE_NAME);
	if (err != 0) {
		dev_err(inst->dev, "unable to allocate device number");
		goto failed_alloc_chrdev;
	}
	cdev_init(&rk3568_gpiomem_cdev, &rk3568_gpiomem_fops);
	rk3568_gpiomem_cdev.owner = THIS_MODULE;
	err = cdev_add(&rk3568_gpiomem_cdev, rk3568_gpiomem_devid, 1);
	if (err != 0) {
		dev_err(inst->dev, "unable to register device");
		goto failed_cdev_add;
	}

	/* Create sysfs entries */

	rk3568_gpiomem_class = class_create(THIS_MODULE, DEVICE_NAME);
	ptr_err = rk3568_gpiomem_class;
	if (IS_ERR(ptr_err))
		goto failed_class_create;

	rk3568_gpiomem_dev = device_create(rk3568_gpiomem_class, NULL,
					rk3568_gpiomem_devid, NULL,
					"gpiomem");
	ptr_err = rk3568_gpiomem_dev;
	if (IS_ERR(ptr_err))
		goto failed_device_create;

	dev_info(inst->dev, "Initialised: Registers at 0x%08lx",
		inst->gpio_regs_phys);

	return 0;

failed_device_create:
	class_destroy(rk3568_gpiomem_class);
failed_class_create:
	cdev_del(&rk3568_gpiomem_cdev);
	err = PTR_ERR(ptr_err);
failed_cdev_add:
	unregister_chrdev_region(rk3568_gpiomem_devid, 1);
failed_alloc_chrdev:
failed_get_resource:
	kfree(inst);
failed_inst_alloc:
	dev_err(inst->dev, "could not load rk3568_gpiomem");
	return err;
}

static int rk3568_gpiomem_remove(struct platform_device *pdev)
{
	struct device *dev = inst->dev;

	kfree(inst);
	device_destroy(rk3568_gpiomem_class, rk3568_gpiomem_devid);
	class_destroy(rk3568_gpiomem_class);
	cdev_del(&rk3568_gpiomem_cdev);
	unregister_chrdev_region(rk3568_gpiomem_devid, 1);

	dev_info(dev, "GPIO mem driver removed - OK");
	return 0;
}

static const struct of_device_id rk3568_gpiomem_of_match[] = {
	{.compatible = "rockchip,rk3568-gpiomem",},
	{ /* sentinel */ },
};

MODULE_DEVICE_TABLE(of, rk3568_gpiomem_of_match);

static struct platform_driver rk3568_gpiomem_driver = {
	.probe = rk3568_gpiomem_probe,
	.remove = rk3568_gpiomem_remove,
	.driver = {
		   .name = DRIVER_NAME,
		   .owner = THIS_MODULE,
		   .of_match_table = rk3568_gpiomem_of_match,
		   },
};

module_platform_driver(rk3568_gpiomem_driver);

MODULE_ALIAS("platform:gpiomem-rk3568");
MODULE_LICENSE("GPL");
MODULE_DESCRIPTION("gpiomem driver for accessing GPIO from userspace");
MODULE_AUTHOR("Luke Wren <luke@raspberrypi.org>");
```

First, the line <span style="{{ site.code }}">inst = kzalloc(sizeof(struct rk3568_gpiomem_instance), GFP_KERNEL);</span>.<br>
The <span style="{{ site.code }}">kzalloc</span> function is similar to <span style="{{ site.code }}">malloc</span>.<br>
Think of it as the kernel version of dynamic memory allocation; <span style="{{ site.code }}">kzalloc</span> also initializes the memory to zero.<br>
<span style="{{ site.code }}">GFP_KERNEL</span> is a flag indicating that memory should only be allocated if it is available.<br>

The statement <span style="{{ site.code }}">inst->dev = dev;</span> connects the dev of the instance defined at the beginning of the code to the dev of pdev.<br>

<br>
This section passes hardware resources to the instance.
```

		dev_err(inst->dev, "failed to get IO resource");
		err = -ENOENT;
		goto failed_get_resource;
	}
```
Maps the register physical address to the resource start address.<br>
The error log ( <span style="{{ site.code }}">ENOENT</span> ) corresponds to "No such file or directory".<br>

Next is the device driver initialization.<br>
Since it's a char device driver, <span style="{{ site.code }}">cdev_init</span> is used for initialization.<br>
Major/minor numbers are also allocated using the <span style="{{ site.code }}">alloc_chrdev_region</span> function.
```
	/* Create character device entries */

	err = alloc_chrdev_region(&rk3568_gpiomem_devid,
				  DEVICE_MINOR, 1, DEVICE_NAME);
	if (err != 0) {
		dev_err(inst->dev, "unable to allocate device number");
		goto failed_alloc_chrdev;
	}
	cdev_init(&rk3568_gpiomem_cdev, &rk3568_gpiomem_fops);
	rk3568_gpiomem_cdev.owner = THIS_MODULE;
	err = cdev_add(&rk3568_gpiomem_cdev, rk3568_gpiomem_devid, 1);
	if (err != 0) {
		dev_err(inst->dev, "unable to register device");
		goto failed_cdev_add;
	}
```
The <span style="{{ site.code }}">cdev_init</span> function takes the target cdev and its ops as parameters.<br>
It uses the <span style="{{ site.code }}">file_operations</span> defined earlier.<br>
<span style="{{ site.code }}">goto</span> is used for error handling.<br>
I won't cover that in detail here.<br>

<span style="{{ site.code }}">goto</span> is used for error handling.<br>
Again, I won't go into detail here.<br>

<span style="{{ site.code }}">remove</span> section:
```
static int rk3568_gpiomem_remove(struct platform_device *pdev)
{
	struct device *dev = inst->dev;

	kfree(inst);
	device_destroy(rk3568_gpiomem_class, rk3568_gpiomem_devid);
	class_destroy(rk3568_gpiomem_class);
	cdev_del(&rk3568_gpiomem_cdev);
```

Removes the allocated memory, device, class, and character device,<br>

Back to the driver registration section.
```
static struct platform_driver rk3568_gpiomem_driver = {
	.probe = rk3568_gpiomem_probe,
	.remove = rk3568_gpiomem_remove,
	.driver = {
		   .name = DRIVER_NAME,
		   .owner = THIS_MODULE,
		   .of_match_table = rk3568_gpiomem_of_match,
		   },
};

module_platform_driver(rk3568_gpiomem_driver);
```
The remaining part is <span style="{{ site.code }}">of_match_table</span>.<br>
The match table is provided by the driver to allow the <span style="{{ site.code }}">device-tree</span> to manage the device.<br>
It's a table provided by the driver.<br>
It uses the <span style="{{ site.code }}">compatible</span> attribute from the tree.<br>
The driver is loaded by comparing the <span style="{{ site.code }}">compatible</span> value in the tree with the values in the driver's match table.<br>
The value is a string.<br>

As for why it's managed this way,<br>
it's a bit off-topic, but in kernel version 2.*.y, the register function was called every time a driver was added.<br>
the register function was called.<br>
However, as the number and variety of devices grew, management became increasingly difficult.<br>
Consequently, when the Linux kernel major version moved to 3, the <span style="{{ site.code }}">device-tree</span> was introduced.<br>
With device drivers now managed via the compatible value in the tree, the concept of a match table was also established.<br>

<br>
In any case, we define the match table this way and register it using a macro.
```
static const struct of_device_id rk3568_gpiomem_of_match[] = {
	{.compatible = "rockchip,rk3568-gpiomem",},
	{ /* sentinel */ },
};

MODULE_DEVICE_TABLE(of, rk3568_gpiomem_of_match);
```
Registers the mapping table using the <span style="{{ site.code }}">MODULE_DEVICE_TABLE</span> macro,<br>
which requires <span style="{{ site.code }}">\<linux/of.h\></span>.<br>
